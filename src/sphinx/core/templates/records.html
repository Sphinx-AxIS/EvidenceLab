Evidence Records

Records are the raw evidence data ingested into this case. Each record is a single event or log entry stored as structured JSON.

Hunt First, Build Second

  • The Analytics page is the primary hunting workspace for finding suspicious patterns and candidate events.
  • The Records page is the detailed evidence browser for opening specific events and inspecting them closely.
  • For Sigma authoring, start with win_evt_* records and use the Windows Channel, EventID, and Search Raw Data filters.
  • Once you find a promising event, open Detail first and use the event context there to decide whether a detection is justified.
  • Rule-building actions appear on the detail page after the event is understandable in context.

Record Types

  • suricata_alert — IDS alerts triggered by Suricata rules against PCAP data.
  • suricata_flow — Network flow metadata from Suricata.
  • zeek_conn — Connection logs from Zeek (duration, bytes, service, state).
  • zeek_dns / zeek_http / zeek_ssl — Protocol-specific Zeek logs.
  • tshark_stream — Reconstructed TCP stream payloads (printable ASCII content).
  • win_evt_* — Windows Event Log records (Security, PowerShell, Sysmon, etc.).
  • vol_* — Volatility memory forensics output (processes, network, handles).

Usage

  • Use the type filter dropdown to narrow by record type.
  • Use Visible Columns to expose additional evidence fields directly in the table, such as task names, usernames, image paths, or EventRecordID values.
  • A fixed bottom scrollbar appears when the table is wider than the viewport so you can scroll across visible columns without jumping to the bottom of the page.
  • Click a record to see its full JSON content and extracted entities.
  • Records are referenced by ID in findings and detection rules for traceability.
